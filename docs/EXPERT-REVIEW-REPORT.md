# 三一学习 Agent 方案 · 专家团评审报告

> 日期：2026-07-19  
> 评审对象：[[quiz-app/docs/AI-LEARNING-AGENT-ROADMAP.md]]  
> 评审团：E5 分析参谋 · E11 架构顾问 · E12 管理学大师 · E9 项目总管

---

## 一、总体评价

四位专家一致认为：**方案方向正确，架构设计合理，Phase 1+2 可行性高**。文件系统即认知载体（Vault-as-Brain）是最大的亮点——轻量、可读、和 Claude Code 同源，避免了过早引入重型基础设施。Skill + Model 双路由架构恰当地分离了"做什么"和"用谁做"，为未来扩展留足空间。

**但有三处结构性风险**：(1) 上下文窗口预算未规划，(2) 并发文件写入无保护，(3) 工时估算偏乐观。

---

## 二、E5 分析参谋 · 评审

### 优点

**1. 度量驱动不足，但方向可补。** 方案中"Agent 认识学生"是一个定性目标，缺乏可量化的成功指标。但这不是设计缺陷——Phase 1 定位为 MVP，定性验收（"朋友用了7天"）是合理的。Phase 2 之后需要补充 KPI：Agent 推荐的薄弱点与用户实际错误的匹配率、用户采纳学习建议的比例等。

**2. 降级策略设计精良。** 模型路由中的降级链（多模态 Key 缺失 → PyMuPDF 文本提取 → DeepSeek Chat）体现了生产级思维。大多数学生项目不做降级，直接报错。这个设计让你的系统即使在用户配置不全时也能提供服务。

### 不足

**1. 文件并发写入风险未覆盖。** 当 Agent 在 `reflect` 中写 mastery.md，同时用户在做下一轮练习触发新的 mastery 更新，两个写操作可能竞态导致数据丢失。SQLite 有 WAL 锁保护，但文件系统没有。

**2. 缺少"AI 记忆衰减"机制。** `.agent/memory.md` 只追加不裁剪，3 个月后会膨胀到难以管理。AI 的上下文窗口有限，旧记忆如果从不清理会挤占新认知的空间。

### 两条建议

> **建议 1**：Phase 1 就加入文件写入锁（`fcntl.flock` 或简单的 `.lock` 文件），所有 vault 写入操作必须先获取锁。这是 20 行代码的事，但能防止未来最难排查的数据错乱。

> **建议 2**：`.agent/memory.md` 增加"记忆权重"字段。Agent 每次追加新条目时标注权重（1-5），每周自动归档权重 ≤2 的旧条目到一个 `memory-archive.md`。这样 AI 上下文里始终是高质量的近期记忆。

## 三、E11 架构顾问 · 评审

### 优点

**1. Vault-as-Brain 架构是正确选择。** 不做重型数据库、不做微服务、不做消息队列——这些决策和 2C2G 服务器约束完美匹配。文件系统即数据库，Markdown 即数据格式，Obsidian 兼容——这套技术选型在"个人开发者 + 小服务器 + 快速迭代"这个约束集下是局部最优解。

**2. Skill prompt 与 Model 路由的解耦设计。** Skill 负责"做什么"（diagnose / study-plan / explain），Model Router 负责"用谁做"（DeepSeek / Kimi / 豆包）。两个路由层独立演进、互不干扰。这在软件架构中叫做"关注点分离"，是成熟的设计模式。

### 不足

**1. 上下文窗口预算未做规划。** DeepSeek Chat 的上下文窗口是 64K tokens。方案中 Agent 每次对话要注入：system.md + memory.md + profile/全部 + knowledge/全部 + 7天 journal + Skill prompt。粗略估算：一个活跃用户 3 个月后，仅 knowledge/ 就可能 5000+ tokens，加上其他文件很容易超 20K tokens。如果还要对话 + 出题 + 解析，余量会非常紧张。

**2. 缺少"认知摘要层"。** Agent 每次都读全部文件，而不是读摘要。更好的做法：每个 vault 目录下有一个 `_summary.md`（由 Agent 自动维护），包含该目录的压缩版内容。Agent 日常只需要读摘要，需要细节时再深入具体文件。

### 两条建议

> **建议 3**：Phase 1 加入 **Context Budget Manager**。一个简单的 Token 计数器，在组装 Agent 上下文时按优先级注入文件：system.md > memory.md > profile/ > knowledge/{当前科目}/ > journal/最近3天 > 其余。超过预算时自动裁剪低优先级文件，并在 prompt 中标注"以下内容因上下文限制未加载：..."。

> **建议 4**：每个 vault 目录引入 `_summary.md`。Agent 每次更新 mastery.md / wrong-book.md / journal 时，同时更新对应目录的 `_summary.md`（不超过 500 字）。日常对话只读摘要，深度诊断（/diagnose skill）时才读完整文件。这相当于给 Agent 加了一层"快速预览"能力。

## 四、E12 管理学大师 · 评审

### 优点

**1. "7天连续使用"作为北极星指标极为精准。** 这个指标同时测量了三个维度：产品价值（用户愿意回来）、学习效果（7天足以看到进步）、技术稳定性（7天不出 bug）。比 DAU、MAU 这类虚荣指标更有意义。建议把这个指标写在 README 第一行。

**2. Vault 作为"学生数字孪生"的概念有长期价值。** 如果用户用了半年，vault 里有 100+ 篇 journal、完整的知识掌握历史、所有错题的模式分析——这就是一份不可替代的个人数据资产。用户迁移成本极高，留存自然高。这是很多教育产品想做但做不到的。

### 不足

**1. 缺少入门引导设计。** 新建用户 vault 是空的，Agent 第一次对话时没有任何认知，会产生"冷启动"问题。方案假设用户会主动填 profile、主动上传资料、主动练习——但现实是大部分用户需要被引导。如果没有好的 onboarding，用户第一次打开网站看到空白页面就走了。

**2. Skill prompt 是"开发者写的"，缺少基于真实数据的迭代机制。** 6 个 Skill 的 prompt 写得很好，但它们是静态的。现实中，"帮我看看哪里弱"这个表述可能匹配不到 `/diagnose` 的触发词（"分析弱点"），导致路由到默认 Agent 而非深度诊断。触发词的覆盖度需要通过真实对话来验证和补充。

### 两条建议

> **建议 5**：设计一个 **Onboarding Wizard（3 步入门）**。用户注册后弹出：(1) 你最近在学什么？（自动建第一个科目）(2) 你的目标是什么？（写入 goals.md）(3) 你喜欢怎么学？（写入 preferences.md）。三步走完，vault 里有基本认知，Agent 第一句对话就是个性化的。全程不到 1 分钟。

> **建议 6**：建立 **Skill 触发词反馈闭环**。记录每次用户消息 → 路由到的 Skill → 用户是否继续追问（说明路由对了）还是换种说法（说明路由错了）。每周手动 review 10 条路由错误的案例，更新触发词列表。Phase 1 的触发词不需要完美，但需要这个迭代机制。

## 五、E9 项目总管 · 评审

### 优点

**1. 阶段划分有清晰的交付物和验收标准。** 每个 Phase 都明确写了"验收"段落，这是项目管理的良好实践。特别是 Phase 1 的验收标准（"朋友注册 → 做练习 → 第二天 Agent 说出昨天错了什么"）是具体的、可测试的、和用户体验直接关联的。

**2. 技术不做过度设计。** 方案里大量"不做的事"（不引入新数据库、不本地跑模型、不过早做多 Agent）体现了工程克制。在资源有限的情况下，决定"不做什么"比"做什么"更重要。

### 不足

**1. 工时估算偏乐观，缺少缓冲。** Phase 1 的 25h 估算是"理想编码时间"。实际开发中，调试 PDF 解析的边界情况、处理文件编码问题、写异常处理——这些"隐形工作"通常占 30-40%。建议所有工时 ×1.3。

**2. Phase 内没有可部署的中间里程碑。** 25h 做一个完整的 Phase 1 意味着在 25h 内没有任何可部署的增量。如果中间遇到阻塞（比如 PDF 解析库的兼容性问题），整个 Phase 都无法交付。应该把 Phase 1 拆成 3 个独立可部署的子阶段。

### 两条建议

> **建议 7**：所有工时 ×1.3。Phase 1: 25h→33h, Phase 2: 20h→26h, Phase 3: 15h→20h。这不是"放水"，是工程现实。如果你提前完成，多出来的时间可以做 polish。

> **建议 8**：Phase 1 拆为 3 个独立可部署的 sprint：
> - Sprint 1（10h）：Vault 底座 — 文件创建、读写、quota 监控。部署后用户可以上传资料，Agent 能读 profile.md。
> - Sprint 2（10h）：Skill 路由 — 6 个 Skill prompt + 路由引擎 + Agent 上下文注入。部署后用户说"帮我诊断"能路由到正确的 Skill。
> - Sprint 3（13h）：模型路由 — 多模型配置 + 降级策略 + PDF 解析。部署后用户可以配 Kimi Key 分析 PDF。
> 每个 sprint 结束时都有可演示的增量。如果 sprint 2 卡住了，至少 sprint 1 的东西已经在生产环境跑着。

## 六、综合评审矩阵

| 维度 | 评分 | 说明 |
|------|------|------|
| **可行性** | 8/10 | 技术栈匹配服务器约束，Phase 1-2 完全可行。Phase 3+ 需要服务器升级。 |
| **稳定性** | 6/10 | 文件系统简单但缺少并发保护。加入文件锁 + context budget 后 → 8/10。 |
| **目标达成度** | 7/10 | Vault + Skill + 自学习闭环如果做出来，确实能"认识学生"。但冷启动和上下文窗口是达成度上限。 |
| **架构优雅度** | 9/10 | Vault-as-Brain + Skill路由 + Model路由 三层解耦，是教科书级的设计。 |
| **可维护性** | 8/10 | Markdown 文件人类可读，调试方便。但文件多了之后需要索引机制。 |
| **可扩展性** | 7/10 | Skill 可插拔、模型可替换，架构扩展性好。但单机部署限制了用户规模上限。 |

**综合评分：7.5/10** — 一个面向个人开发者 + 小服务器的务实方案。8 条建议若全部采纳，可提升至 8.5/10。
