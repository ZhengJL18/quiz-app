## Round 1：Squad 分裂 · 三视角独立评审

### E5-α · 分析视角（数据与度量）

**结论：方案度量化程度大幅提升，但缺少用户侧的"感受度量"。**

加分项：
- Context Budget Manager 的 token 预算策略明确（30K 上限，优先级排序），可量化验证。
- 记忆权重归档（weight 1-5 + 周归档）提供了清晰的数据生命周期管理。
- Sprint 拆分后每个 sprint 有独立验收标准，比之前的"模糊 25h"好太多。

不足：
- **缺少用户感受指标**。方案度量了系统的健康度（token 预算、磁盘配额、文件锁），但没有设计如何度量"用户觉得 Agent 多了解自己"。建议 Phase 2 加入：用户打开首页时 Agent 建议的采纳率、用户主动追问的比例（追问多=Agent 没讲清楚）。
- **Onboarding 的完成率未追踪**。3 步向导如果用户在第 2 步放弃，vault 是半空状态。需要监控每一步的流失率。

### E5-β · 性能视角（服务器与成本）

**结论：方案在 2C2G 服务器上可行，但 PDF 解析是隐形炸弹。**

加分项：
- PyMuPDF 作为多模态降级方案是聪明选择，避免了对昂贵 API 的强制依赖。
- Context Budget Manager 保护了 API 调用成本（不用每次都发 50K tokens 的 prompt）。
- 模型路由的降级链设计降低了"单点 API 挂了整个系统崩"的风险。

不足：
- **PDF 解析在 2C2G 上可能 OOM**。PyMuPDF 解析一个 50MB 的扫描版 PDF 可能吃 500MB+ 内存。方案缺一个"PDF 大小限制"或"解析队列"——大 PDF 应该异步处理并通知用户结果，而不是阻塞 Agent 对话。
- **Context Budget 的策略可能需要动态调整**。方案中"30K tokens 硬上限"在 DeepSeek Chat（64K 窗口）里留了一半余量，但 DeepSeek Reasoner 的窗口可能不同。应该做成环境变量配置而非硬编码。

### E5-γ · 用户体验视角

**结论：Onboarding 解决了冷启动，但日常使用的"Agent 存在感"还需要加强。**

加分项：
- 3 步 Onboarding 设计精良，每步都有一个明确的 vault 产出。
- Skill 路由让用户用自然语言触发深度功能，降低了学习成本。
- Phase 1 验收标准里"Agent 第一句话就是个性化的"是很好的 UX 目标。

不足：
- **Agent 的"主动性"不足**。方案中 Agent 主要响应用户请求（对话、诊断、出题），但没有设计"Agent 主动触达"的场景。比如：用户 3 天没登录 → Agent 发一条推送"你上次的换元积分还没掌握，回来练 5 道吧"。这在 Phase 1 不需要，但应该在 Phase 2 的 journal 自动生成中埋点。
- **Skill 触发失败的用户体验未定义**。如果用户说"帮我看看哪里不行"，Skill 路由匹配不到任何触发词，会走默认 Agent 行为——用户感觉"Agent 没听懂"。应该有一个 fallback："我不确定你想做哪种分析，你想让我帮你：A.诊断薄弱点 B.制定计划 C.生成报告？"

## Round 1：相互评价

### [E5-α → E5-β] 互评

1. 事实准确 [通过]：β 对 PDF 解析的内存评估准确，PyMuPDF 确实有 OOM 风险。
2. 逻辑完整 [存疑]：β 提到"做成环境变量配置"解决 Context Budget 硬编码问题，但这会产生配置漂移——不同用户的 Budget 不同可能导致体验不一致。建议 Budget 策略基于模型类型自适应而非环境变量。
3. 互补性 [α 的补充]：β 没提到多模态 API 的成本。一张图片走 Kimi 多模态的成本是纯文本的 5-10 倍。建议 Phase 1 的模型路由中加一个"月度 API 成本预估"给用户看到。
4. 改进建议：P1 — 在 PDF 解析模块加文件大小限制（>20MB 异步处理 + 通知用户）。

### [E5-α → E5-γ] 互评

1. 事实准确 [通过]：γ 对 Onboarding 和 Skill 路由的分析与方案一致。
2. 逻辑完整 [存疑]：γ 的"Agent 主动触达"建议是 Phase 2 的内容，γ 标注了"Phase 1 不需要"但放在了 Phase 1 的评审里，可能造成混淆。
3. 互补性 [α 的补充]：γ 的 fallback 建议（"你想让我帮你 A/B/C？"）非常实用。应该写进 Skill 路由的默认行为，而不仅是 UX 层的考虑。
4. 改进建议：P0 — Skill 路由匹配失败时，输出结构化选项（A/B/C）而非静默降级到默认 Agent。

### [E5-β → E5-α] 互评

1. 事实准确 [通过]：α 指出的"缺少用户感受指标"正确。方案目前只有技术指标。
2. 逻辑完整 [通过]：Onboarding 流失率监控确实应该在 Phase 1 埋点。
3. 互补性 [β 的补充]：α 的用户感受指标可以用最简单的方式实现——Phase 2 的 journal 中加一行"今日 Agent 建议是否采纳？[是/否]"。
4. 改进建议：P1 — journal 末尾加采纳率标记。

### [E5-β → E5-γ] 互评

1. 事实准确 [通过]：γ 的 UX 分析覆盖了关键场景。
2. 逻辑完整 [通过]：Skill fallback UI 是 Phase 1 应该解决的问题。
3. 互补性 [β 的补充]：γ 的"Agent 主动性"不只在推送层面。Agent 应该学会在对话中插入"顺便一提"——比如用户问导数问题，Agent 说"答完了。顺便，你上周极限错了 3 题，要不要趁热打铁？"
4. 改进建议：P1 — Agent system prompt 中加入"顺便一提"行为模式。

### [E5-γ → E5-α] 互评

1. 事实准确 [通过]：α 的分析客观，没有过度解读。
2. 逻辑完整 [通过]：Onboarding 监控 + 用户感受指标形成了完整的度量链。
3. 互补性 [γ 的补充]：α 的数据指标可以设计得更轻——不一定要埋点系统。Phase 1 可以做最简单的：Onboarding 完成率 = vault 里有 profile 的用户数 / 总注册数。Mastery 更新率 = 本周 mastery.md 被更新过的用户数 / 活跃用户数。
4. 改进建议：P1 — 用 vault 文件时间戳做最轻量的指标监控。

### [E5-γ → E5-β] 互评

1. 事实准确 [存疑]：β 说 PDF 解析可能 OOM，但 PyMuPDF 的默认行为是 lazily load pages，不是一次性加载全部。50MB PDF 通常不会导致 500MB 内存占用。但 β 关于"大文件异步处理"的建议仍然正确。
2. 逻辑完整 [通过]：Context Budget 动态配置的需求确实存在。
3. 互补性 [γ 的补充]：除了 PDF 大小限制，还应该加一个"解析进度反馈"——大 PDF 解析时 Agent 说"你的教材有 200 页，正在解析中，预计 2 分钟完成"。
4. 改进建议：P1 — PDF 解析加进度反馈。

## Round 2：对抗双审

### E7-α · 独立审校（事实 + 逻辑）

> ⚠️ 隔离上下文：E7-α 只看到了方案文件，未看到 Round 1 的 Squad 评审。

**事实准确性审核**：

1. ✅ Vault 目录结构定义清晰，文件类型和用途一致。
2. ✅ Skill 路由规则表覆盖了主要场景，触发词设计合理。
3. ✅ 模型路由规则表中的降级策略完整且可执行。
4. ⚠️ Context Budget Manager 的 30K token 硬上限：DeepSeek Chat 的窗口确实是 64K，但实际可用略少（system prompt + 输出 token 都要占）。30K 预留了约 25K 给输出，合理。但 DeepSeek Reasoner（如有）窗口可能不同，建议验证。
5. ⚠️ 文件锁使用了 `fcntl.flock`——这在 Linux 上正确，但 Windows 开发环境不支持。方案标注了"跨平台 fallback"，但未说明 fallback 方案。建议明确：Windows 上用 `msvcrt.locking` 或创建 `.lock` 文件 + `os.path.exists` 检查。

**逻辑完整性审核**：

1. ✅ Phase 1 → Phase 2 的递进关系清楚。Phase 1 建基础设施，Phase 2 做自学习闭环。
2. ✅ Sprint 拆分粒度合理：Sprint 1（Vault+Onboarding）独立可部署，Sprint 2（Skill）依赖 Sprint 1 的 Vault，Sprint 3（Model）依赖 Sprint 2 的上下文组装。
3. ⚠️ Phase 2 的自学习 Prompt 模板中，`{session_file 内容}` 可能很大（一轮练习的完整记录）。没有提到如何截断或摘要。建议在 Context Budget Manager 中同样应用于 reflect 调用。
4. ❌ 缺少 **阶段性回滚策略**。如果 Sprint 2 上线后发现严重 bug，能否独立回滚到 Sprint 1？还是必须整体回退？建议每个 Sprint 的 API 变更做到向后兼容。

**审查结论**：PASS（有条件），上述 ⚠️ 和 ❌ 标记的问题需要在开发中解决。

### E7-β · 独立审校（表达 + 完整性）

> ⚠️ 隔离上下文：E7-β 只看到了方案文件，未看到 Round 1 的 Squad 评审。

**表达清晰度审核**：

1. ✅ 方案结构层次分明：架构理念 → Agent 设计 → Skill 模块 → 知识层 → 练习层 → 日志层 → 实施计划 → 对标。读者可以按顺序理解。
2. ✅ Vault 目录树用 ASCII art 呈现，直观清晰。
3. ✅ Skill prompt 示例（diagnose.md / study-plan.md）足够详细，开发者可以直接照着写。
4. ⚠️ Phase 1 的 Sprint 3（模型路由）描述相对简略，只有 3 行。建议补充具体的 ModelRouter 匹配逻辑伪代码，和其他 sprint 保持一致的详细度。
5. ⚠️ Context Budget Manager 的优先级排序公式未给出。当前描述"system.md > memory.md > profile/ > ..."是定性描述，建议给一个伪代码实现。

**完整性审核**：

1. ✅ 覆盖了从用户注册到 Agent 自学习的完整链路。
2. ✅ 存储预算、服务器压力、技术风险评估都做了。
3. ⚠️ 缺少 **API 版本管理策略**。Phase 1 有大量的 API 变更（AI 助教接口的上下文注入改为 vault 读取）。如果前端还是旧版 API 会怎样？建议加入 API 版本号（如 `/api/v2/` 或请求头版本协商）。
4. ⚠️ 缺少 **数据迁移计划**。现有用户的数据在 SQLite 里，Phase 1 后要迁移到 vault 文件系统。方案说"数据库改动：零"，但实际上现有数据（practice_sessions、wrong_book、chapter_mastery）需要从 SQLite 导出到 vault。这不是"零改动"。
5. ❌ 缺少 **用户 vault 的备份策略**。每个用户 ≤1GB 的数据集，如果服务器磁盘损坏，所有 Agent 记忆会丢失。建议在 Phase 1 加入简单的 vault 备份（定时 rsync 到另一个目录，或利用现有 VPS 快照）。

**审查结论**：PASS（有条件），建议补充 API 版本管理、数据迁移计划和 vault 备份策略。
